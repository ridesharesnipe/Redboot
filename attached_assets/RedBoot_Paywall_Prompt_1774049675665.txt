PAYWALL IMPLEMENTATION — RED BOOT'S SPELLING ADVENTURE
=======================================================

IMPORTANT: Do not change any existing UI, styling, components, or functionality. 
Only ADD the paywall system described below. Do not refactor or "improve" anything else.

This is a Stripe-powered paywall for the web app. Not Apple IAP.


== OVERVIEW ==

The app currently lets users go from onboarding → child setup → dashboard → unlimited practice for free.

We are adding a paywall that gates everything AFTER the first practice session.

The user flow becomes:

1. Onboarding cards (existing — no changes)
2. Child setup — name + grade (existing — no changes)  
3. Landing + character select (existing — no changes)
4. Dashboard — upload word list (existing — no changes)
5. ONE free practice session (existing — no changes to the practice screen itself)
6. ★ NEW: "Aha moment" paywall appears when the first session ENDS
7. ★ NEW: Everything is locked after that until they subscribe
8. ★ NEW: Ongoing paywall placements for returning non-subscribers

No paywall appears before the first practice session. The user experiences the app first, THEN we ask for money.


== SUBSCRIPTION STATE ==

Add a simple subscription state to localStorage:

Key: "redboot-subscription"
Value: JSON object:
{
  "status": "free" | "trial" | "active" | "cancelled",
  "freeSessionUsed": false | true,
  "stripeCustomerId": null | "cus_xxx",
  "subscriptionId": null | "sub_xxx", 
  "currentPeriodEnd": null | "2026-04-20T00:00:00Z",
  "plan": null | "monthly" | "annual"
}

On first install, status is "free" and freeSessionUsed is false.

After the first practice session completes (when SimplePractice shows the results screen), set freeSessionUsed to true.


== PAYWALL GATE LOGIC ==

Check subscription state before allowing access to:
- SimplePractice (practice sessions) — allow if freeSessionUsed is false OR status is "trial"/"active"
- FridayTest — locked unless status is "trial" or "active"
- FridayTestSimulator — locked unless status is "trial" or "active"  
- ParentAnalytics — locked unless status is "trial" or "active" (show blurred preview with subscribe button overlay)
- TreasureRoad — locked unless status is "trial" or "active"
- SeaMonsterBattle — locked unless status is "trial" or "active"

The dashboard, word list upload, treasure vault, badge gallery, and onboarding remain free.

== STRIPE INTEGRATION ==

Use Stripe Checkout (hosted) for payment processing. This keeps PCI compliance simple.

Products to create in Stripe:
- Monthly: $6.99/month with 7-day free trial
- Annual: $39.99/year with 7-day free trial  
- Monthly (no trial): $6.29/month, no trial
- Annual (no trial): $35.99/year, no trial
- Annual (abandonment offer): $24.99/year with 7-day free trial

Server endpoints needed:

POST /api/create-checkout-session
  Body: { plan: "monthly" | "annual", trial: true | false, offer: "standard" | "abandonment" }
  Returns: { sessionId, url }
  Creates a Stripe Checkout session with the appropriate price and redirects user to Stripe.

POST /api/webhook (Stripe webhook endpoint)
  Handles: checkout.session.completed, customer.subscription.updated, customer.subscription.deleted
  On checkout.session.completed: 
    - Store stripeCustomerId and subscriptionId
    - Set status to "trial" (if trial) or "active" (if no trial)
    - Set currentPeriodEnd
    - Set plan to "monthly" or "annual"
  On customer.subscription.updated:
    - Update status and currentPeriodEnd
  On customer.subscription.deleted:
    - Set status to "cancelled"

GET /api/subscription-status
  Returns current subscription state for the user
  (Since we don't have auth, tie this to a device identifier stored in localStorage — 
   generate a UUID on first visit, store as "redboot-device-id", send with all API calls)

Success/Cancel URLs for Stripe Checkout:
  Success: /dashboard?subscription=success (show a brief "Welcome aboard, captain!" confirmation)
  Cancel: /dashboard (triggers transaction abandonment logic on next action)
